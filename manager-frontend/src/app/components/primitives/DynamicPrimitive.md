# Dynamic UI Primitives

An ontology-driven system for recursive user interface generation and complex data management.

## 1. Motivation: Why Dynamic Primitives?

In scientific data management, records are often hierarchical and strictly defined by an ontology. Rather than building static forms for every possible data type, the **Dynamic Primitive** system allows the UI to adapt automatically to any ontology structure.

Key use cases include:
- **Data Entry Forms**: Given an ontology structure, the `DynamicPrimitiveComponent` can display data in the appropriate primitive component in the user interface.
- **Workflow Inputs**: Capturing specific parameters required for cloud-based tasks or simulations where the input schema might change based on the selected method.
- **Semantic Consistency**: Ensuring that data entered in the UI matches the semantic constraints and types defined in the backend ontology.

## 2. Primitive Catalog and Properties

The system currently supports 13 specialized primitive components. Each component receives an `OntologyStructure` object which defines how it should behave and render.

### Origin of the Structure
The `OntologyStructure` is not hardcoded in the frontend. It is dynamically generated by the backend on request. When the frontend needs to render a specific ontology class, it makes a **POST** request to the following endpoint:

`POST /api/datastore/ui-template`

The backend processes this request using the `DocumentTemplateForUI` class, which traverses the ontology definition and generates the metadata required for the UI. 

> [!NOTE]
> For a deep dive into the backend generation logic and code snippets, see the [Ontology Structure Generation](./StructureGeneration.md) guide.

### The `OntologyStructure` Interface
All primitives rely on this metadata to determine their role. This interface is defined in `base-primitive.ts`:

```typescript
export interface OntologyStructure {
  identifier: string;    // Unique ID for the field
  classname: string;     // The ontology class name
  label?: string;        // Human-readable label
  comment?: string;      // Descriptive text (tooltips/placeholders)
  
  // Logic Flags
  isObject?: boolean;    // If true, contains nested properties
  isArray?: boolean;     // If true, is a list of values
  
  // Primitive Type Selection (Booleans)
  isClassification: boolean;
  isParagraph: boolean;
  isOneLine: boolean;
  isEmail: boolean;
  isURL: boolean;
  isBoolean: boolean;
  isKeyword?: boolean;
  isKeywordSet: boolean;
  isFileSource: boolean;
  
  // Additional Constraints & Nesting
  choices?: any;         // Optional allowed values
  properties?: { [key: string]: OntologyStructure }; // Child properties for isObject
}
```

### Available Primitives
| Primitive | Ontology Flag | Description |
|-----------|---------------|-------------|
| **OneLine** | `isOneLine` | Standard short text strings. |
| **Paragraph** | `isParagraph` | Long descriptions or abstracts. |
| **Boolean** | `isBoolean` | Binary flags (true/false). |
| **Keyword** | `isKeyword` | Individual categorical terms. |
| **Email/URL** | `isEmail`/`isURL` | Specialized validated strings. |
| **Classification** | `isClassification` | Hierarchy-based selections. |
| **File Source**| `isFileSource`| External data file links. |
| **JSON Object**| `isObject` | Recursive container for nested data. |
| **JSON Array** | `isArray` | Recursive container for lists. |

## 3. Input/Output Model: Detailed Example

The system relies on a strict two-way data flow via Angular Inputs and Outputs. Every primitive extends `BasePrimitiveComponent`.

### Data Propagation (Parent to Child)
The orchestrator passes the `OntologyStructure` and the current `value` as `@Input()` properties.

### Data Bubbling (Child to Parent)
When a user interacts with a primitive (e.g., typing in a text field), the change is captured and emitted upwards.

#### 1. Template Interaction (`oneline-primitive.ts`)
The template uses `ngModel` for binding and `(ngModelChange)` to trigger the update:
```html
<mat-form-field appearance="outline" style="width: 100%;">
  <mat-label>{{ structure.label || structure.classname }}</mat-label>
  <input matInput 
         [ngModel]="value" 
         (ngModelChange)="updateValue($event)" 
         [placeholder]="structure.comment || ''">
</mat-form-field>
```

#### 2. The `updateValue` Emit logic (`base-primitive.ts`)
The `updateValue` method is implemented in the abstract base class to ensure consistent behavior across all primitives:
```typescript
@Component({ template: '' })
export abstract class BasePrimitiveComponent {
  @Input() value: any;
  @Input() structure!: OntologyStructure;
  @Output() valueChange = new EventEmitter<any>();

  updateValue(v: any) {
    if (this._value !== v) {
      this._value = v;                // Update internal state
      this.valueChange.emit(v);      // Bubbe change to orchestrator
    }
  }
}
```

## 4. Recursive Flow: Property Categorization

The system builds its hierarchy through the `JsonObjectPrimitiveComponent`. During initialization, it transforms the ontology's `properties` dictionary into a sorted collection for the UI. Specifically, the initialization code fills these four category elements:

- **`gridProps`**: Populated with small categorical fields (Keywords, Booleans, etc.) for high-density grid rendering.
- **`wideProps`**: Populated with full-width identification fields like `isOneLine` and `isFileSource`.
- **`contentProps`**: Populated with large multi-line text fields (typically `isParagraph`).
- **`recursiveProps`**: Populated with nested Objects and Arrays to trigger the next level of the DFS building process.

### Categorization Logic (TypeScript)
In `ngOnInit`, we create `allProps` and filter them into these four distinct visual buckets:

```typescript
if (this.structure.properties) {
  // Convert dictionary into map-able array
  const allProps: PropertyItem[] = Object.keys(this.structure.properties).map(key => ({
    key,
    structure: this.structure.properties![key]
  }));

  // Iterate and Sort into logic buckets
  allProps.forEach(prop => {
    const s = prop.structure;
    if (s.isArray || s.isObject) {
      this.recursiveProps.push(prop); 
    } else if (s.isParagraph) {
      this.contentProps.push(prop);
    } else if (s.isFileSource || s.isOneLine) {
      this.wideProps.push(prop);
    } else {
      this.gridProps.push(prop);
    }
  });
}
```

### Visual Categories (Template Logic)

The template iterates over these pre-sorted arrays, passing each item back to the `DynamicPrimitiveComponent` for specialized rendering.

#### Category 1: Grid Primitives
Used for high-density metadata like Keywords, Booleans, and Classifications.
```html
<div *ngIf="gridProps.length > 0" class="props-grid">
  <div *ngFor="let prop of gridProps; trackBy: trackByKey" class="property-cell">
    <app-dynamic-primitive 
      [structure]="prop.structure" 
      [value]="value[prop.key]"
      (valueChange)="updateProperty(prop.key, $event)">
    </app-dynamic-primitive>
  </div>
</div>
```
*   **Logic**: Loops through `gridProps` (populated in the TS code) and renders them in a CSS grid.

#### Category 2: Identification & Full-Width Titles
Used for `isOneLine` fields or file identifiers.
```html
<div *ngIf="wideProps.length > 0" class="props-wide">
  <div *ngFor="let prop of wideProps; trackBy: trackByKey" class="property-row">
    <app-dynamic-primitive 
      [structure]="prop.structure" 
      [value]="value[prop.key]"
      (valueChange)="updateProperty(prop.key, $event)">
    </app-dynamic-primitive>
  </div>
</div>
```
*   **Logic**: Loops through `wideProps` and ensures they occupy the full width of the parent container.

#### Category 3: Large Content Blocks
Used for `isParagraph` fields level fields.
```html
<div *ngIf="contentProps.length > 0" class="props-content">
  <div *ngFor="let prop of contentProps; trackBy: trackByKey" class="property-row">
    <app-dynamic-primitive 
      [structure]="prop.structure" 
      [value]="value[prop.key]"
      (valueChange)="updateProperty(prop.key, $event)">
    </app-dynamic-primitive>
  </div>
</div>
```
*   **Logic**: Loops through `contentProps`, typically appearing after identifying metadata.

#### Category 4: Recursive Structures (Complex Types)
Used for nested Objects and Arrays.
```html
<div *ngIf="recursiveProps.length > 0" class="props-recursive">
  <div *ngFor="let prop of recursiveProps; trackBy: trackByKey" class="property-row">
    <app-dynamic-primitive 
      [structure]="prop.structure" 
      [value]="value[prop.key]"
      (valueChange)="updateProperty(prop.key, $event)">
    </app-dynamic-primitive>
  </div>
</div>
```
*   **Logic**: Loops through `recursiveProps` to trigger further levels of the DFS building process. By placing this last, the UI reflects the natural nesting order of the ontology.

## 5. Orchestration: DynamicPrimitiveComponent

The `DynamicPrimitiveComponent` acts as a smart dispatcher. It evaluates the `OntologyStructure` provided by the backend and decides which specific component to instantiate.

### Dispatch Logic
The component template uses `*ngIf` guards to switch between types:
```html
<div class="dynamic-wrap">
  <!-- Recursion Case: Objects and Arrays -->
  <app-json-object-primitive *ngIf="structure.isObject" 
    [structure]="structure" [value]="value" 
    (valueChange)="onChildValueChange($event)">
  </app-json-object-primitive>

  <!-- Leaf Case: Specific Primitives -->
  <app-oneline-primitive *ngIf="structure.isOneLine" 
    [structure]="structure" [value]="value"
    (valueChange)="onChildValueChange($event)">
  </app-oneline-primitive>
  ...
</div>
```

## 6. Developer Guide: Adding a New Primitive

To add a new primitive type (e.g., a `DatePrimitive`):

1. **Create the Component**:
   - Generate a new component folder (e.g., `date-primitive`).
   - Extend `BasePrimitiveComponent`.
   - Implement `getData()` and `setData()`.
2. **Update the Interface**:
   - In `base-primitive.ts`, add the new flag to `OntologyStructure` (e.g., `isDate: boolean`).
3. **Register the Component**:
   - Import the new component in `dynamic-primitive.ts`.
   - Add it to the `imports` array of the `@Component` decorator.
4. **Update the Dispatcher**:
   - Add a new `*ngIf` block in `dynamic-primitive.ts`'s template:
     ```html
     <app-date-primitive *ngIf="structure.isDate" 
       [structure]="structure" [value]="value" 
       (valueChange)="onChildValueChange($event)">
     </app-date-primitive>
     ```

Implementation

The ExtractDescription Class

The core logic is implemented in the ExtractDescription service. This class serves as the bridge between raw user input and the Discovery Engine API.

Method Breakdown: extract(String prompt, String datatype)

This method orchestrates the call to the LLM agent. It uses SearchServiceClient from the Google Cloud Discovery Engine Library.

Resource Path Breakdown

The servingConfig string is a fully qualified resource name. It tells the Google Cloud API exactly which infrastructure to use for the retrieval process.

Component Role
projects/ The Google Cloud Project ID (e.g., blurock-database). This defines the billing and IAM context.
locations/ The geographic Data Residency Location (typically global).
collections/ The Data Store Collection (usually default_collection). Collections allow you to group multiple data stores and engines under a single logical unit.
engines/ The unique Agent ID (or Engine ID). This is the cerebral center that connects your prompt to the underlying data stores.
servingConfigs/ The Serving Configuration (typically default_search), which defines the ranking, filtering, and summarization behavior.
Configuration

The Role of the Agent ID

The Agent ID (Engine) is the most critical component in the RAG pipeline. While a Data Store holds the raw documents, the Agent provides the intelligence to search, summarize, and interpret queries.

Connecting Data to Intelligence

To set up an Agent ID in the Google Cloud Console:

  1. Navigate to Vertex AI Search & Conversation.
  2. Create a New App of type Search.
  3. Connect one or more Data Stores to this app.
  4. Once created, the Engine ID (which we call Agent ID) is visible in the App Configuration settings.

Why Link a Data Store?

Linking a Data Store to the Agent is what transforms a generic AI model into a specialized scientific assistant. The Data Store acts as the "External Memory" or **Ground Truth** for the Agent.

1. What information is provided?

The Data Store provides the agent with the specific Dictionary Terms and Semantic Descriptions extracted from our formal ontologies. This includes project-specific categories (like catalogobjects) and scientific definitions that the base LLM may not have encountered during its initial training.

2. How the Agent uses it (The RAG Flow)

When a user submits a prompt, the Agent performs a two-step process:

  • Retrieval: It searches the linked Data Store for the most relevant documents (terms) that match the user's intent.
  • Augmentation: It "stretches" the prompt by injecting the retrieved scientific context into the hidden instructions sent to the LLM.

The Agent doesn't just "guess" a title; it looks at similar items in our database to ensure the generated Title and Keyword align with our existing scientific indexing standards.

3. Risk of Omission: What if we didn't link a Data Store?

If the Data Store were omitted, the system would revert to a generic LLM search. This leads to several critical failures in a scientific context:

  • Hallucinations: The model might invent keywords that don't exist in our project's ontology.
  • Lack of Domain Precision: It might use common English meanings for terms that have very specific, technical definitions within JThermodynamicsCloud.
  • Context Blindness: The model would be unable to distinguish between different datatypes, resulting in metadata that cannot be used for faceted browsing or structured searches.
Generative AI

Understanding Generative Summaries

Our implementation leverages the Generative Search capabilities of Vertex AI. According to the official Google documentation on Getting Search Summaries, this feature allows the agent to synthesize a response based on the top search results, a process commonly known as Grounding.

Grounded Summarization Logic

Unlike a traditional search that returns a list of links, a generative response uses an LLM to "read" the results and generate a new, concise output. In our system, we use this to transform a multi-page raw description into a strictly formatted JSON title and keyword. This is configured via the SummarySpec and ModelPromptSpec parameters.

Request Configuration (JSON)

The following JSON object represents the ContentSearchSpec we send to the API. It tells the agent to use the top 5 documents as context and apply our custom instructions.

"contentSearchSpec": {
  "summarySpec": {
    "summaryResultCount": 5,
    "modelPromptSpec": {
      "preamble": "ROLE: You are a Content Summarization Agent..."
    },
    "ignoreAdversarialQuery": true,
    "includeCitations": false
  }
}

Response Metadata (JSON)

The resulting SearchResponse includes a unique summary block. The summaryText field contains the generated JSON string that we then sanitize and parse in Java.

"summary": {
  "summaryText": "{\"title\": \"Heat Capacity of Propane\", \"keyword\": \"HeatCapacity\"}",
  "summarySkippedReasons": []
}

By setting summaryResultCount to 5, we ensure the agent has enough scientific context to be accurate, but remains focused enough to provide a concise summary rather than an essay.

Prompting

Model Instruction (The Preamble)

The PREAMBLE is the foundational instruction set for the LLM. It defines the agent's identity and strict operational constraints. 

private static final String PREAMBLE = """
    ROLE: You are a Content Summarization Agent.
    TASK: Analyze the USER QUERY and provide a concise title and a single keyword.
    
    RULES:
    1. TITLE: Create a brief, descriptive title...
    2. KEYWORD: Provide a single keyword... (e.g., 'HeatCapacityCalculation').
    3. OUTPUT: Return ONLY a valid JSON object. No markdown, no explanations.
    
    FORMAT: {"title": "Summary Title", "keyword": "KeyWord"}
    
    USER QUERY: {query}
    """;

By explicitly defining the RULES and FORMAT, we minimize the risk of non-JSON output, which is critical for automated downstream processing.

API Usage

Customizing the SearchRequest

The SearchRequest is configured to trigger the summarization model rather than just standard retrieval. 

SearchRequest request = SearchRequest.newBuilder()
    .setServingConfig(servingConfig)
    .setQuery(searchQuery)
    .setFilter(filter)
    .setPageSize(10)
    .setContentSearchSpec(SearchRequest.ContentSearchSpec.newBuilder()
        .setSummarySpec(SearchRequest.ContentSearchSpec.SummarySpec.newBuilder()
            .setSummaryResultCount(5)
            .setModelPromptSpec(
                SearchRequest.ContentSearchSpec.SummarySpec.ModelPromptSpec.newBuilder()
                    .setPreamble(PREAMBLE)
                    .build())
            .build())
        .build())
    .build();
  • setContentSearchSpec: Signals that we want an AI-generated summary across results.
  • setSummaryResultCount(5): Ground the AI in the top 5 most relevant documents found in the data store.
  • setModelPromptSpec: Injects our custom PREAMBLE to guide the summary.
Data Handling

Parsing the Agent Output

The response from Discovery Engine contains a Summary object. The summaryText field contains the JSON string we requested.

Response Sanitization

// 1. Get raw text from the summary field
String rawJson = response.getSummary().getSummaryText();

// 2. Clean markdown artifacts (e.g. ```json blocks)
String cleanJson = rawJson.replaceAll("```json|```", "").trim();

// 3. Extract purely the JSON block
int start = cleanJson.indexOf("{");
int end = cleanJson.lastIndexOf("}");
if (start >= 0 && end > start) {
    cleanJson = cleanJson.substring(start, end + 1);
}

Note: We distinguish between Structured Data (the raw fields found in documents) and Summary Text (the natural language output generated by the LLM grounded in those documents).

Firestore Schema & Sessions

A technical look at multi-user isolation, session lifecycle management, and global document discovery using Collection Groups.

The Concept of a "Session"

In this architecture, a **Session** represents a unique, scoped instance of a user's interaction (typically tied to a single browser tab). This is critical for scientific workflows where a user might be performing multiple independent calculations simultaneously.

Why distinct sessions?

  • Tab Isolation: Prevents "cross-talk" where Tab A accidentally picks up results intended for Tab B.
  • Contextual Grouping: All artifacts (molecules, results, logs) for one specific experiment are grouped under a single parent.
  • Resource Management: Provides a natural hook for automated cleanup of abandoned or "dead" sessions.

Ontology-Driven Hierarchy

The Firestore path is determined dynamically. While the top levels handle multi-tenancy, the collection names themselves are derived from the ontology.

users / {uid} // Global User Insulation
sessions / {sessionId} // Browser Tab Isolation
lastActivity: Timestamp // For cleanup logic
molecule-tasks / {taskId} // Sub-collection
calculations / {calcId} // Result storage

Global Discovery via Indexing

Because tasks are nested deep within user/session paths, the backend cannot listen to a single collection(). Instead, it uses a **Collection Group Query**.

What is a Collection Group?

It allows you to query all collections with a specific name (e.g., all molecule-tasks sub-collections) regardless of where they live in the hierarchy. This is how the Java backend "sees" every pending task across all users.

Required Indexing: Firestore requires an explicit index for any collection group query that involves filters. Without this index, the backend listener will fail to start.

Deep Dive: The Mechanics of Group Indexing

By default, Firestore indexes are scoped to a single collection. To query across multiple collections of the same name, a Collection Group Index is mandatory. This creates a global, flattened view of all matching collections.

Why is it mandatory?

Firestore's performance guarantee (O(1) search time) relies on pre-computed indexes. Searching every sub-collection in the database at runtime would violate this principle. The group index tells Firestore to maintain a dedicated internal table for that specific collection ID globally.

Implementation

Frontend: Path Building (TS)
// session.service.ts
getCollectionPath(colName: string): string {
  const uid = this.auth.currentUser.uid;
  const sid = this.sessionId;
  return `users/${uid}/sessions/${sid}/${colName}`;
}

// submission
const path = sessionService.getCollectionPath('molecule-tasks');
await addDoc(collection(firestore, path), taskData);
Backend: Global Listener (Java)
// Use collectionGroup for global discovery
Query tasks = firestore.collectionGroup("molecule-tasks");

tasks.whereEqualTo("status", "pending")
     .addSnapshotListener((snapshots, e) -> {
         for (DocumentSnapshot doc : snapshots) {
             // doc.getReference().getPath() 
             // identifies the specific user/session
             processTask(doc);
         }
     });

Session Lifecycle Management

To avoid "dead sessions," a scheduled task in the backend periodically prunes inactive branches of the tree.

Backend: SessionCleanupTask.java
@Scheduled(fixedDelay = 86400000) // Daily
public void cleanup() {
  Date cutoff = new Date(System.currentTimeMillis() - 24hr);
  
  // Find sessions with no activity in 24 hours
  Query old = firestore.collectionGroup("sessions")
                       .whereLessThan("lastActivity", cutoff);

  for (DocumentSnapshot doc : old.get()) {
      deleteRecursive(doc.getReference());
  }
}

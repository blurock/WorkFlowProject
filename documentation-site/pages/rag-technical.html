Implementation Details

A technical guide for software engineers on the end-to-end flow of the Retrieval-Augmented Generation system.

Stage 1

Ontology Term Extraction

The process begins by traversing the formal scientific ontology to identify terms that should be searchable by the RAG system. This is primarily handled by CreateDataStoreTermSet.

Logical Flow

The addToDataStoreTermSet method performs a recursive descent into ontology classes. It specifically targets subclasses of dataset:ChemConnectPrimitiveDataStructure. 

// Extraction logic from CreateDataStoreTermSet.java
public static void addToDataStoreTermSet(DictionaryTermSet terms, Set<String> processed, String classname, String datatype) {
    JsonObject template = CreateDocumentTemplate.createTemplateWithAnnotations(classname);
    JsonObject annotations = template.getAsJsonObject("annotations");
    
    for (String key : annotations.keySet()) {
        if (OntologyUtilityRoutines.isSubClassOf(key, "dataset:ChemConnectPrimitiveDataStructure", false)) {
            // Extract identifier, label, and description
            JsonObject element = annotations.getAsJsonObject(key);
            String comment = element.get(AnnotationObjectsLabels.comment).getAsString();
            DictionaryTerm term = new DictionaryTerm(name, key, comment, datatype);
            terms.addTerm(term);
        }
    }
}

The CreateDocumentTemplate class extracts Labels, Identifiers, and Comments (which serve as the semantic description) from the OWL ontology files, converting them into a JSON structure for easy processing.

Stage 2

Data Modeling for Semantic Context

Extracted terms are encapsulated into standard POJOs (Plain Old Java Objects) to ensure strict typing and metadata consistency. This structure is critical for Retrieval-Augmented Generation as it defines the "ground truth" the LLM will reference.

DictionaryTerm Structure

  • ID: The formal ontology URI (e.g., dataset:ChemSubstance), acting as the unique document key.
  • Term: Human-readable label (e.g., "Chemical Substance") used for user-facing results.
  • Description: The semantic heart of the term. This natural language string is what the embedding model processes for vector search.
  • Metadata: Contains the datatype (e.g., catalogobjects), enabling faceted filtering during retrieval.
Stage 3

Persistence to Cloud Firestore

The system persists terms to Google Cloud Firestore. To handle thousands of ontology terms efficiently, we use a high-throughput initialization and batch strategy.

Initialization & Batched Writes

The FirestoreClient is initialized using Application Default Credentials (ADC). We then use WriteBatch to commit terms in chunks of 500, which is the Firestore atomic limit. 

// Firestore Initialization (via Firebase Admin SDK)
Firestore db = FirestoreClient.getFirestore();
CollectionReference collection = db.collection("dictionary_terms");

// Merge-on-Write Strategy in WriteDictionaryTerms.java

2. Configure Schema & Indexing

The schema defines how Discovery Engine interprets your Firestore documents. Marking fields as Indexable allows them to be used in filters, while Searchable makes them part of the vector search space.

// Explicit DataStore Schema Definition
{
  "id": "string",          // Unique formal URI
  "term": "string",        // Human label
  "description": {         // Semantic source
    "type": "string",
    "searchable": true,    // Required for Vector Search
    "indexable": true
  },
  "metadata": {
    "datatype": {
      "type": "array",
      "indexable": true,   // Required for filtering
      "retrievable": true
    }
  }
}

Stage 5

Retrieval via Discovery Engine

The final step is the runtime retrieval where the user query is matched against the indexed terms.

1. Constructing the ServingConfig

The ServingConfig is the fully qualified resource name that identifies your Project, Data Store instance, and the specific Search Configuration to use. 

// Resource format for ServingConfig
String servingConfigS = String.format(
    "projects/%s/locations/%s/collections/%s/engines/%s/servingConfigs/%s",
    "blurock-database",    // GCM Project ID
    "global",               // Location
    "default_collection",   // Collection ID
    "analyseterms_...",     // Engine (Agent) ID
    "default_search"        // Serving Config ID
);

2. Building the SearchRequest

The SearchRequest defines what we are looking for and how results should be restricted. The query parameter shown below is the natural language prompt provided by the user to be interpreted using the RAG method. 

SearchRequest request = SearchRequest.newBuilder()
    .setServingConfig(servingConfigS)
    .setQuery(query) // The natural language prompt from the user
    .setFilter("metadata.datatype: ANY(\"" + datatype + "\")")
    .setPageSize(6)
    .build();

.setQuery(query): This is the prompt provided by the user. Discovery Engine converts this text into a vector embedding and performs a similarity search against the description field in the index.

.setFilter(...): This ensures total scientific precision. Even if a term like "Thermodynamic" exists in multiple datasets, the ANY() filter restricts the vector search to only those documents that belong to the specified datatype (e.g., catalogobjects).

3. Parsing the SearchPagedResponse

When searchServiceClient.search(request) returns, the result is an encapsulated SearchPagedResponse. The most critical part is extracting the Structured Data, which is a key-value mapping of your original Firestore fields.

Actual JSON Data

{
  "id": "dataset:ChemSubstance",
  "term": "Chemical Substance",
  "metadata": {
    "datatype": ["catalogobjects"]
  }
}

Java Access Logic

// Extracting from Map<String, Value>
Map<String, Value> fields = result.getDocument()
    .getStructData().getFieldsMap();

// 1. Simple String Access
String id = fields.get("id").getStringValue();

// 2. Nested Struct Access (metadata)
Struct meta = fields.get("metadata").getStructValue();

// 3. Array Access (datatype)
ListValue types = meta.getFieldsMap().get("datatype")
    .getListValue();
String firstType = types.getValues(0).getStringValue();

Derived Data & Ranking

Beyond the raw fields, Discovery Engine provides "Derived Data"—system-generated scores that are essential for RAG confidence filtering.
